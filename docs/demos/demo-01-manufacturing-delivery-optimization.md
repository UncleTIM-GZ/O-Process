# Demo 1: 制造企业订单交付优化 — 流程诊断与改进

> **场景:** 一家 500 人的中型制造企业，订单交付周期比竞争对手长 30%，销售与生产跨部门协作摩擦大，流程文档存在但无人执行。

---

## 一、SCOR 框架定位：问题归属哪个流程域？

调用流程知识库，检索 SCOR 五大域（Plan / Source / Make / Deliver / Return）在 PCF 中的集成节点：

**Tool:** `search_process`

```json
{ "query": "Plan Deliver supply chain", "lang": "en", "limit": 10 }
```

**框架定位结果:**

| SCOR 域 | 对应 PCF 节点 | 与本案关联度 |
|---------|--------------|-------------|
| **Plan** | `4.1.9` 规划供应链 / `4.1.11` 规划生产 / `4.1.12` 规划交付 | **高** — 交付周期长的根源往往在计划协同 |
| **Deliver** | `4.4.5` 交付库存产品 / `4.4.6` 交付按订单生产的产品 | **高** — 直接影响交付周期 |
| **Make** | `4.3.6` 按订单生产 / `4.3.7` 按订单设计 | **中** — 生产排程影响交期 |
| **Enable** | `13.1` 管理业务流程 / `13.10` 持续改进 | **高** — 流程文档无人执行属于治理失效 |

**诊断结论:** 本案横跨三个 SCOR 域 —— **Plan（计划协同断裂）**、**Deliver（交付执行低效）**、**Enable（流程治理缺失）**。

## 二、PCF 对标：标准流程编号与最佳实践

### 问题 1: 订单交付周期比竞争对手长 30%

**Tool:** `get_process_tree`

```json
{ "process_id": "4.1", "max_depth": 2 }
```

**对标流程:**

| PCF 编号 | 流程名称 | 最佳实践建议 |
|----------|----------|-------------|
| `4.1.2` | 管理产品需求 | 建立需求预测机制，MAPE < 25% |
| `4.1.4` | 创建和管理主生产计划 | MPS 与销售订单联动，周级滚动更新 |
| `4.1.12` | 规划交付 (SCOR: Plan Deliver) | 交付承诺基于产能实际约束，非销售口头承诺 |
| `3.5.4` | 管理销售订单 | 订单录入即触发产能检查，ATP/CTP 自动校验 |
| `4.4.6` | 交付按订单生产的产品 (SCOR: Deliver MTO) | 端到端订单可视化追踪 |

**关键 KPI（来自知识库）:**

| KPI | 单位 | 行业基准 |
|-----|------|----------|
| 客户订单周期天数 | days | 制造业中位数: 7-14 天 |
| 交付周期天数 | days | 应逐季度下降 |
| 现金到现金周期天数 | days | 越短越好，反映供应链效率 |
| 月度平均预测误差 (MAPE) | % | 目标 < 25% |

### 问题 2: 销售与生产跨部门推责

**Tool:** `search_process`

```json
{ "query": "sales operations planning demand", "lang": "en", "limit": 5 }
```

**对标流程:**

| PCF 编号 | 流程名称 | 最佳实践建议 |
|----------|----------|-------------|
| `4.1.9` | 规划供应链 (SCOR: Plan Supply Chain) | 建立 S&OP 会议机制，销售/生产/物流月度对齐 |
| `4.1.2` | 管理产品需求 | 销售提供需求预测，生产反馈产能约束 |
| `3.5.4` | 管理销售订单 | 订单承诺前必须通过产能校验 |
| `4.1.14` | AI 驱动的需求预测 | 用历史数据替代人工拍脑袋预测 |

**根因分析:** 销售与生产推责的本质是 **信息断裂** —— 销售承诺交期时不知道产能约束，生产排程时不知道订单优先级。缺少 `4.1.9 规划供应链` 这个协同节点。

## 三、流程诊断摘要

### 问题定位

| 维度 | 诊断 | SCOR 域 | PCF 对标 |
|------|------|---------|----------|
| 交付周期长 | 订单-生产-交付链路无端到端协同 | Plan + Deliver | `4.1.4`, `4.1.12`, `4.4.6` |
| 跨部门推责 | 缺少 S&OP 机制，信息在销售与生产间断裂 | Plan | `4.1.9`, `4.1.2`, `3.5.4` |
| 流程文档空转 | 无流程 Owner，无绩效度量，无执行约束 | Enable | `13.1.1`, `13.1.4`, `13.10` |

### 根因假设

```
                    ┌─────────────────┐
                    │  流程治理缺失   │ ← 根因 (Enable)
                    │ 13.1.1 / 13.1.4 │
                    └────────┬────────┘
                             │
               ┌─────────────┴─────────────┐
               ▼                           ▼
    ┌──────────────────┐       ┌──────────────────┐
    │  S&OP 机制缺失   │       │  流程文档空转     │
    │  4.1.9 / 4.1.2   │       │  13.1.2 / 13.1.5 │
    └────────┬─────────┘       └──────────────────┘
             │
    ┌────────┴─────────┐
    ▼                  ▼
┌────────┐      ┌──────────┐
│销售盲目│      │生产被动排│
│承诺交期│      │程，频繁插│
│ 3.5.4  │      │单 4.1.4  │
└────────┘      └──────────┘
        │              │
        └──────┬───────┘
               ▼
    ┌──────────────────┐
    │ 交付周期长 30%    │ ← 表象
    │ 4.4.6 / 4.1.12   │
    └──────────────────┘
```

**核心判断:** 交付周期长是 **果**，不是因。真正的根因是 **流程治理缺失（13.1）** 导致 **计划协同断裂（4.1.9）**，进而造成销售与生产各自为战。

### 推荐改进路径（3 步）

| 步骤 | 行动 | 对标流程 | 预期效果 | 周期 |
|------|------|----------|----------|------|
| **Step 1** | **建立 S&OP 月度会议机制** — 销售提供滚动 3 月需求预测，生产反馈产能约束，共同确认订单优先级和可承诺交期 | `4.1.9` 规划供应链 + `4.1.2` 管理产品需求 | 消除信息断裂，交期准确率提升 | 1-2 月落地 |
| **Step 2** | **订单流程系统化** — 销售录入订单时系统自动进行 ATP（可用量承诺）检查，超产能订单需生产确认后方可承诺 | `3.5.4` 管理销售订单 + `4.1.4` 主生产计划 | 杜绝「销售乱承诺」，减少插单率 | 2-3 月实施 |
| **Step 3** | **指定流程 Owner + KPI 考核** — 每个关键流程指定负责人，将「订单交付周期」和「预测准确率」纳入销售和生产的共同 KPI | `13.1.1` 流程治理 + `13.1.4` 流程绩效 | 流程从纸面走向执行，持续改进闭环 | 1 月启动，持续运行 |
